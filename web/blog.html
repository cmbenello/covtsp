18h45m → 18h05m: Optimizing the Tube Challenge Solver

The London Tube Challenge asks you to visit all 272 Underground stations as fast as possible. The current Guinness World Record is 17h46m. Our automated solver does it in 18h05m. Here's how we shaved 40 minutes off the initial solution.

The problem

The solver builds a time-expanded graph (TEG) from TfL's GTFS timetable data. Every node is a (station, departure_time) pair. Edges are train legs, waiting arcs, and walking transfers. For London on a weekday, this gives us 160K nodes and 1.26M edges.

Horizontal arrows = wait arcs (stay at station). Diagonal edges (not shown) = train legs and walking transfers between stations.
The solver finds the shortest path that touches at least one node per required station.

The task is a Covering TSP: visit at least one node per required station, minimizing total elapsed time. No return to start. NP-hard in general. We use a greedy heuristic (nearest-unvisited in time-space) and compute an LP relaxation lower bound to prove solution quality.

The baseline solver—k=1 nearest-neighbor from Ealing Broadway at 06:33—produced a valid 272/272 route in 18h45m. Not bad. But the Guinness record is 17h46m, and we knew there was room to improve.

The Kensington Olympia breakthrough

The single biggest optimization came from one station: Kensington Olympia.

K.O. is served by a shuttle from Earl's Court, and it only runs 9 times per day: 5 morning shuttles (05:49–07:20) and 4 evening shuttles (20:01–20:40). That's it. A 13-hour dead zone in the middle of the day with zero service.

Only 9 TEG nodes for the entire day. Miss both windows and you can't complete the challenge.

The old solver started at 06:33 and worked greedily through the network. By the time it reached Earl's Court, it was 20:41—one minute after the last K.O. shuttle. The solver had to backtrack through a long detour to catch it, adding 60+ minutes.

The fix: force the solver to visit K.O. first thing in the morning, before running the greedy nearest-neighbor. Start at Finchley Road at 05:41, take the Metropolitan line to Earl's Court, catch the 05:49 shuttle to K.O., and then continue the greedy solve from there. 40 minutes saved.

Urgency scoring

K.O. isn't the only time-constrained station. Mill Hill East is at the end of a single-track branch with limited service. The standard greedy solver often defers it until too late, missing it entirely (271/272).

We added an urgency score that biases the nearest-neighbor toward stations whose service windows are closing. At each step, the solver weighs: 

score = travel_time + urgency_weight * time_until_last_service(station)

Stations with early last-service times get a bonus. The urgency weight has a sharp phase transition: from Euston Square, urgency < 0.16 misses Mill Hill East (271/272), while urgency ≥ 0.16 catches it (272/272) at a cost of ~21 extra minutes. From Finchley Road, the urgency weight doesn't matter—the route naturally covers Mill Hill East regardless.

There's a sharp tradeoff between coverage and speed. Dropping just one station (271/272) saves 20–40 minutes, because the last few stations are always the hardest to reach—they're on branch ends with infrequent service.

What didn't work

We tried a lot of things that didn't improve the solution:

  • k=3 lookahead: Consistently misses 5 branch-end stations. The 2-step penalty causes the solver to defer branches it should visit early.
  • Line-aware scoring: Biasing toward same-line stations never achieves 272/272.
  • TEG local search (2-opt/or-opt): Random moves on a 272-station route find no improvement. The search space is too large and the objective too noisy (time-dependent edges mean small reorderings can cascade).
  • Forced visit windows: Broad windows break other coverage; narrow windows don't trigger when needed.
  • 500-trial randomized search: Random perturbations from the top starting stations couldn't find any 272/272 solutions with K.O. forced visit.

The fundamental issue is that local search doesn't work well on time-expanded graphs. Swapping two stations in the visit order changes the arrival time at every subsequent station, which changes which trains are available, which changes everything downstream. The landscape is rugged.

Running transfer filtering

Beyond riding trains, Tube Challenge competitors can run between stations. When you're stuck at the end of a branch, it's sometimes faster to run to a nearby station on a different line than to backtrack through the network.

But computing accurate running times requires the Google Routes API. With 272 stations, all-pairs queries would cost ~$375. We need a way to figure out which pairs are actually worth computing.

Filter 1: Distance cutoff

Only consider pairs within 20km Haversine distance. Beyond that, running would take 2+ hours at race pace and will almost never beat the train. This cuts from 37K to 21K pairs. Not enough.

Filter 2: Branch terminus stations

Running between stations is only useful at the ends of lines. If you're mid-line, you can just take the train. Running only becomes competitive when you're at a terminus and the nearest unvisited station is faster to reach on foot than by backtracking through the network.

We identify terminus candidates by reconstructing trip sequences from GTFS data: the last 3 stops of each branch end. We keep pairs where at least one station is a terminus candidate and they're on different lines. This drops to 10,921 pairs.

Filter 3: Train dominance prune

The aggressive filter. For each remaining pair (A, B):

Even if you could run in a perfectly straight line at maximum sprint speed—ignoring streets, crossings, and navigation—would running still be slower than the train's raw travel time (ignoring headway waits)?
Dominance condition
crow_flies_time = haversine(A, B) / sprint_speed
train_time = shortest_path(A, B) // no waits, just travel

if crow_flies_time ≥ train_time:
    eliminate // running can never win

Both sides are optimistic lower bounds. If the train's lower bound beats running's lower bound, the dominance holds for all real values.

This eliminates 93% of remaining pairs. Result: 723 pairs at ~$7 API cost. A 98% reduction from the naive approach, without losing any pair that could actually improve the route.

What's next

The solver currently runs 19 minutes behind the human world record. The main gaps:

  • LP relaxation lower bound (Phase 3): needed to compute the actual optimality gap and prove solution quality.
  • Running transfers: integrating the 723 filtered pairs into the TEG as running edges could shave significant time off branch-end detours.
  • Better search: simulated annealing over the solver parameters (start station, urgency weight, injection thresholds) rather than greedy search.
  • Multi-city: NYC (475 stations, 21h35m), Berlin U-Bahn (170 stations, 7h53m), and more.

The code is open-source. Add your city by dropping a GTFS feed and a YAML config.
