19 Minutes from the World Record with a Greedy Heuristic

The Guinness record for visiting all 272 London Underground stations is 17h46m. An automated solver on public timetable data does it in 18h05m. This is how we got there—and where the remaining gap lives.

Covering TSP on a time-expanded graph

The Tube Challenge isn't standard TSP. It's harder in three ways: no return to start, time-dependent edges (you can only board when a train departs), and covering variant (visit at least one node per station, not every node). This is a Covering TSP on a time-expanded graph. NP-hard in general.

In a static graph, the edge A→B costs 3 minutes regardless of when you arrive. In reality, arriving at 8:07 with the next train at 8:12 means waiting 5 minutes—real cost is 8 minutes. A TEG makes this explicit: every node is a (station, departure_time) pair.

Each circle is a (station, time) node. Train legs go diagonally down-right (time advances, station changes). Wait arcs go horizontally (time advances, same station). Walking transfers skip stations. London TEG: 160K nodes, 1.26M edges from 272 stations.

The solver finds the minimum-time path that touches at least one node per required station. We use Dijkstra with early termination (~0.02s per step) on this graph.

The Kensington Olympia breakthrough

The single biggest optimization—40 minutes—came from one station. Kensington Olympia is served by a shuttle from Earl's Court that runs 9 times per day: 5 morning (05:49–07:20), 4 evening (20:01–20:40). That's 9 TEG nodes for the entire day. Compare to Oxford Circus with hundreds.

The greedy heuristic can't see 14 hours ahead. It optimizes locally, and by the time it reaches Earl's Court it's 20:41—one minute after the last shuttle. Cue a 60+ minute detour.

The fix is a prefix strategy: force K.O. first, before the greedy solver runs. This is a form of hard station scheduling—identify bottleneck stations with constrained service windows and handle them explicitly.

Urgency scoring and phase transitions

K.O. isn't the only bottleneck. Mill Hill East (single-track Northern line branch, limited evening service) and Heathrow T4 (separate Piccadilly loop) are also fragile. The solver detects these automatically by analyzing TEG sparsity.

We bias the nearest-neighbor with an urgency term: 

score = travel_time + urgency_weight * time_until_last_service(station)

The weight has a sharp phase transition. From Euston Square:

From Finchley Road the urgency weight doesn't matter—the route naturally covers Mill Hill East regardless. The solver's best variant: K.O. prefix + k=5 NN + urgency 0.5 + T4 injection via Hatton Cross.

What didn't work

  • k=3 lookahead—consistently misses 5 branch-end stations. Multi-step penalty defers branches the solver should visit early.
  • TEG local search (2-opt/or-opt)—finds no improvement. Swapping two stations cascades through the entire route: different arrival times → different available trains → different downstream path. The landscape is rugged.
  • Line-aware scoring—30% same-line bonus never achieves 272/272.
  • 500-trial randomized search—random perturbations from top starts couldn't find 272/272 with K.O. forced.
  • Evening K.O. injection—arrives Earl's Court 20:41, last shuttle 20:40. One minute.

The fundamental issue: local search doesn't work on TEGs. Small reorderings cascade because arrival times shift, changing which trains are available everywhere downstream. Classical TSP moves assume edge weights are fixed. Here they're not.

Running transfer filtering: $375 → $7

Tube Challenge competitors run between stations—it's sometimes faster to run from a branch end to a nearby station on a different line than to backtrack. Computing running times needs the Google Routes API. All-pairs for 272 stations: 37,128 queries, ~$375.

The dominance prune

For each candidate pair (A, B):

crow_flies = haversine(A, B) / sprint_speed  // optimistic: straight line
train_time = shortest_path(A, B)  // optimistic: no headway wait

if crow_flies ≥ train_time: eliminate  // running can never win

Both sides are lower bounds. If the train's lower bound beats running's lower bound, the inequality holds for all real values. The prune is provably correct—it cannot discard a useful shortcut. Eliminates 93% of remaining pairs.

What's next

  • LP relaxation on TEG—current bound (7h43m, static graph) is loose. TEG-based LP would be much tighter but the instance is 160K nodes.
  • Running edges—integrate the 723 filtered pairs into the TEG. Likely where most of the 19-minute gap to the world record lives.
  • Simulated annealing—over solver params (start station, urgency weight, injection thresholds) rather than hand-tuning.
  • More cities—Tokyo, Chicago, Singapore. Each has different topology characteristics that stress different parts of the algorithm.
